    ConfigEnv¶

    背景¶

    当前 WarmPool 主要用于提供可快速复用的运行环境，但在实际使用中，很多 Agent 任务并不只依赖基础镜像本身，还依赖额外的环境配置。例如数据库配置文件、连接凭证、缓存参数、应用运行时配置等。这类配置如果无法在 WarmPool 创建时一并提供，用户就需要在环境启动后再手动补充，既增加使用成本，也会影响环境的一致性和可复用性。

    对于需要依赖数据库或其他基础服务的场景，这一问题尤为明显。以 MySQL 为例，用户在启动对应的 WarmPool 时，通常不仅需要拉起服务本身，还需要同时提供数据库配置文件，并将其挂载到容器指定目录。只有这样，数据库才能在启动时加载正确配置，并在 WarmPool 就绪后立即被 Agent 使用。

    需求目标¶

    为了解决上述问题，WarmPool 需要具备环境配置能力，使用户可以在创建 WarmPool 时，同时声明运行所需的配置资源以及对应的注入方式。系统应在 WarmPool 启动阶段自动完成这些配置资源的准备，并将其挂载或注入到容器中，确保相关服务或 Agent 在启动完成后即可直接使用所需环境。

    该能力的目标包括：

    • 支持在创建 WarmPool 时统一声明所需环境配置。
    • 支持系统自动准备并注入相关配置资源。
    • 确保配置在容器启动前已经生效。
    • 确保 Agent 或依赖服务在环境启动后即可直接使用，无需额外初始化操作。

    使用方式¶

    用户在创建 WarmPool 时，只需要同时描述两件事：

    • 运行所需的配置资源。
    • 这些资源应该如何注入到容器中。

    配置资源通常有两种使用方式：

    • 以文件形式挂载到指定目录，例如数据库配置文件、应用配置文件、证书文件。
    • 以环境变量形式注入到容器中，例如数据库密码、访问凭证、连接参数。

    对于 MySQL 这类服务，常见做法是把数据库配置文件挂载到服务启动目录，同时将敏感信息注入为环境变量。这样 WarmPool 启动完成后，服务就已经具备完整的运行条件。

    核心诉求¶

    用户希望将运行环境本身与其依赖配置统一描述，而不是将配置准备、资源创建、挂载注入等操作拆散到 WarmPool 创建之外的其他手工流程中。

    换言之，用户创建的不是一个“只有基础镜像”的 WarmPool，而是一个“带有完整运行环境约束”的 WarmPool。对于需要特定配置才能正常工作的服务，系统应保证其在 WarmPool 创建完成时已经处于可用状态。

    典型场景¶

    MySQL 场景¶

    当用户创建一个带有 MySQL 环境的 WarmPool 时，需要同时提供 MySQL 所需的配置内容，例如连接数、字符集、数据库初始化参数等。系统应自动准备这些配置，并将其挂载到 MySQL 容器所需目录中，使数据库在启动时即可加载正确配置。

    在这种情况下，WarmPool 就绪后应具备以下特征：

    • MySQL 服务已经启动并加载完成预期配置。
    • Agent 可以直接访问数据库，无需额外配置步骤。
    • 用户不需要手动预创建配置资源，也不需要在容器启动后再次处理挂载问题。

    通用依赖环境场景¶

    除 MySQL 外，该能力也适用于其他依赖配置资源的环境，例如 Redis、Postgres、带凭证的外部服务接入场景，或需要通过配置文件驱动启动行为的应用组件。无论具体服务类型如何，用户的核心诉求都是一致的：在创建 WarmPool 时一次性描述依赖环境，并由系统自动完成环境准备。

    校验要求¶

    为了保证环境在启动时可用，系统需要在创建和更新 WarmPool 时对 ConfigEnv 做校验。校验重点包括：

    • 配置资源名称必须唯一，不能在同一个 WarmPool 内重复。
    • 注入目标必须明确，且目标容器必须真实存在。
    • 挂载目录、环境变量名和资源键名必须合法，不能为空。
    • 模板中引用的变量必须已定义，不能引用未知变量。
    • 不能出现同一个容器内的挂载路径冲突或环境变量冲突。
    • 敏感信息必须按安全方式处理，不能以明文方式暴露在状态或日志中。
    • 配置资源必须能在 WarmPool 启动前准备完成，否则该 WarmPool 不应被视为就绪。

    如果任一项校验失败，系统应直接拒绝生效，并返回明确的错误原因，避免生成一个半配置状态的运行环境。

    预期结果¶

    引入 ConfigEnv 能力后，WarmPool 应从“提供基础运行容器”扩展为“提供可直接使用的完整运行环境”。用户在创建 WarmPool 时即可同时定义运行镜像和所需配置，系统负责将这些配置准备到位，使依赖环境在 WarmPool 启动完成后即处于可用状态。

    这一能力将显著降低带依赖环境任务的使用门槛，提升 WarmPool 的环境一致性、可复用性和开箱即用程度。
